# 准确性校验规则

确保生成的索引信息准确无误，在写入文件前进行多层校验。

## 校验流程

```
元数据校验 → AI 标注校验 → 重复检测 → 文件完整性校验 → 用户确认 → 写入
```

任何一步校验失败，都必须停止流程并向用户报告问题。

## 1. 元数据校验

在 Step 2 收集完用户元数据后立即执行。

### 1.1 配置匹配校验

| 字段 | 校验规则 | 失败处理 |
|------|---------|---------|
| department | 必须在 `config.yaml` 的 `departments[].name` 中存在 | 提示用户选择已有部门或先更新配置 |
| section | 必须在该 department 下的 `sections[].code` 中存在 | 提示用户选择已有科室或先更新配置 |
| access_level | 必须在 `config.yaml` 的 `access_levels` 列表中存在 | 提示用户选择有效权限级别 |

### 1.2 源文件校验

| 检查项 | 规则 | 失败处理 |
|--------|------|---------|
| 文件存在 | 指定路径的文件必须存在 | 提示用户检查路径 |
| 文件格式 | 必须为 `.md` 文件 | 提示仅支持 Markdown |
| 文件非空 | 文件内容不为空 | 提示文件为空，无法分析 |

### 1.3 Domain 名称校验

| 检查项 | 规则 |
|--------|------|
| 合法字符 | Domain 目录名仅允许: 小写字母、数字、连字符 |
| 名称转换 | 用户输入的名称自动转为 slug（如 "Wellness Program" → "wellness-program"） |
| 唯一性 | `knowledge-base/` 下不可有同名目录（除非是添加到已有 domain） |

## 2. AI 标注校验

在 Step 3 生成标注后、Step 4 展示给用户前执行。

### 2.1 关键词回溯验证

对每个生成的关联词汇执行回溯：

```
对于每个关键词:
  1. 在源文档中搜索该词汇（精确匹配或包含匹配）
  2. 如果找到 → 记录出处（章节/段落编号）
  3. 如果未找到 → 标记为 [推断] 并说明推断依据
  4. [推断] 标记的词汇不可超过总数的 30%
```

如果超过 30% 的关键词是推断的，向用户警告标注可靠性较低。

### 2.2 标注完整性检查

所有四项标注字段都必须有内容，不可留空：

| 字段 | 最低要求 |
|------|---------|
| 文档用途 | 至少 1 句话（20 字以上） |
| 关联词汇 | 至少 3 个词汇 |
| 适用场景 | 至少 1 个场景 |
| 适用人群 | 至少 1 个角色 |

### 2.3 标注一致性检查

- 用途描述中提到的关键概念应在关联词汇中出现
- 适用人群应与部门/科室信息逻辑一致
- 适用场景应与文档内容的专业领域匹配

## 3. 重复检测

在 Step 5 写入前执行。

### 3.1 文件名重复

检查 `knowledge-base/{domain}/docs/` 目录下是否已存在同名文件。

```
如果同名文件存在:
  1. 读取已有文件的 frontmatter
  2. 比较 source 字段
  3. 如果 source 相同 → 提示"该文档已索引"，询问是否更新
  4. 如果 source 不同 → 提示文件名冲突，建议重命名
```

### 3.2 内容重复

检查 index.md 中是否已有关联词汇高度重叠的条目。

```
对于 index.md 中的每个现有条目:
  1. 提取其关联词汇列表
  2. 与新文档的关联词汇计算重叠度
  3. 如果重叠度 > 70% → 警告可能是重复文档
  4. 向用户展示可能重复的条目，请求确认
```

### 3.3 索引条目重复

检查 index.md 中是否已有该文件名的条目（防止重复追加）。

## 4. 文件完整性校验

在文件生成后、写入前执行。

### 4.1 Frontmatter 校验

对每个生成的文件检查：

| 检查项 | 规则 |
|--------|------|
| YAML 语法 | frontmatter 必须是合法 YAML |
| 必填字段 | 所有必填字段（见 output-templates.md）都有值 |
| 字段类型 | 日期字段格式为 YYYY-MM-DD，数值字段为数字 |
| 值一致性 | 同一文档在 Level 2 和 Level 3 中的元数据一致 |

### 4.2 跨层级一致性

```
domain.md 的 indexes 列表 ⊇ 实际存在的 index.md
index.md 的 doc_count == docs/ 下的实际文件数
index.md 中每个条目的 filename 都能在 docs/ 下找到对应文件
docs/ 下每个文件的 domain 字段 == domain.md 的 name 字段
```

### 4.3 原始内容保真

Level 3 文档中，frontmatter 之后的内容必须与源文件完全一致（逐字符比较）。

## 5. 用户确认机制

### 确认展示格式

以结构化格式展示所有待写入信息，明确区分"用户提供"和"AI 生成"部分：

```
=== 文档索引预览 ===

【用户提供的元数据】 ✓ 已确认
- 领域: {domain}
- 文件: {filename}
- 权限: {access_level}
- 部门: {department}
- 科室: {section}
- 发起人: {creator}

【AI 生成的业务标注】 ← 请审核
- 用途: {annotation_purpose}
- 关键词汇: {annotation_keywords}
- 适用场景: {annotation_scenarios}
- 适用人群: {annotation_audience}

【校验结果】
- ✅ 元数据校验通过
- ✅ 关键词回溯验证: {found}/{total} 个词汇在文档中找到出处
- ✅ 无重复文档
- ✅ 文件完整性校验通过

请确认以上信息，或告知需要修改的内容。
```

### 修改流程

用户指出修改项 → 更新对应字段 → 重新展示 → 再次确认 → 循环直到用户满意。

## 6. 错误恢复

### 写入失败

如果写入过程中发生错误（如磁盘满、权限不足）：
1. 报告具体错误信息
2. 列出已成功写入和未成功写入的文件
3. 提供手动补救指引

### 部分写入

如果只有部分文件写入成功：
1. 已写入的 Level 3 文档保留
2. 未更新的 index.md 需手动追加条目
3. 提供需要追加的条目内容供用户复制
